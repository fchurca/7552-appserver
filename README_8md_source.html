30 ## Building Docker Image
31 From the root of the project, where the build scripts are located, execute:
32 ```
33 ./appserver.build.sh
34 ```
35 This script builds a Docker image from the Dockerfile in the root of the project's tree. The image will be called fiuba/appserver:dev, and this is the name that the execution scripts will later use.
36 
37 ## Deploying Containers for Local Development
38 
39 First create a Docker network for Mongo and server containers:
40 ```
41 docker network create taller
42 ```
43 Launch now a MongoDB container:
44 ```
45 ./mongo.run.sh
46 ```
47 After Mongo is up, run in another terminal:
48 ```
49 ./appserver.run.sh
50 ```
51 
52 The script launches the image fiuba/appserver:dev inside a container. The container is configured as follows:
53 * Port 8080 of the container is mapped to port 8080 of the host
54 * AppServer container is placed inside network `taller`, with name `appserver.taller`
55 * The project's source code directory in the host is mapped to the directory where gunicorn looks for the application scripts. Since gunicorn is executed with live reload, local live edits to the application code will be instantly reloaded by gunicorn without having to build the image and reload the container once again.
56 * MongoDB container is placed inside network `taller`, with name `mongodb.taller`
57 * By default, the container will try to contact the SharedServer at `sharedserver.taller`
58 More elaborate setups may arise. In that case, refer to the following sections.

65 ## Configuration
66 The following environment variables need to be set:
67 * `APPSERVER_CFG` Location for configuration files, relative to the root of the project.
68 * `SHAREDSERVER_URL` SharedServer URL (default, variable name in `$APPSERVER_CFG/sharedserver.ini`)
69 * `SHAREDSERVER_TOKEN` Initial SharedServer token
70 * `MONGODB_URI` MongoDB URI (default, variable name in `$APPSERVER_CFG/mongo.ini`)
71 * `FCM_API_KEY` Firebase Cloud Messaging key
72 
73 ### SharedServer
74 The variable that defines the URL of the SharedServer instance is defined in `$APPSERVER_CFG/sharedserver.ini`:
75 ```
76 [Connection]
77 sharedservervar=SHAREDSERVER_URL
78 ```
79 Alternatively, the URL itself can be defined in the configuration file:
80 ```
81 [Connection]
82 url=http://sharedserver.taller:5000/api/
83 ```
84 In both cases, the token will be defined as the environment variable `SHAREDSERVER_TOKEN`.
85 
86 ### MongoDB
87 The variable that defines the connection string to some MongoDB instance can be defined in `$APPSERVER_CFG/mongo.ini`:
88 ```
89 [Connection]
90 mongovar=MONGODB_URI
91 database=db
92 ```
93 Then, in this case, the connection string is equal to the value of the variable `MONGODB_URI`. Alternatively, the connection string can be placed directly inside the configuration file as follows:
94 ```
95 [Connection]
96 uri=<connection string>
97 database=db
98 ```
99 For security reasons, however, this may not be desirable.
100 
101 ### Logging
102 Logging is configured in `$APPSERVER_CFG/mongo.ini`. Example:
103 ```
104 [Logging]
105 level=DEBUG
106 format=[%%(asctime)s] [%%(process)d] [%%(levelname)s] %%(name)s - %%(message)s
107 ```
108 For more information regarding Python 3 logging and logging formats, see: https://docs.python.org/3/library/logging.html

133 ## Future improvements
134 In order to make this daemon bulletproof, several improvements can be made. Namely:
135 * Use of USIG can be replaced by calls to Google or OpenStreetMap in order to infer street addresses outside of Buenos Aires.
136 * A back office module can be implemented, or an administration API to be exposed to the SharedServer backoffice.
137 * The position of the passenger is tracked during the trip in order to calculate distance. However, the raw positions as informed by the passenger are used. This can result in unexpectedly high costs if the positions jump around. The datapoints can be smoothed out with LOESS and outlier filtering, either by local variance or by position error. Or with LOWESS weighted by (π/2)-arctan(error). Or some nonlinear regression algorithm or other. Or something.
138 * The deployment methods described use a static number of worker processes. Means to ensure an on-demand elastic deployment of daemons can be realised.
139 * Even more testing. There's never enough testing.

148 ## Troubleshooting
149 ### Mongo complains about `No space left on device: "/data/db/journal"`
150 Dangling stale volumes may still be laying around after an untidy exit, such as a power loss.
151 The following commands for listing and removing volumes may require root permissions.
152 In that case, use `su` or `sudo` with the usual precautions. You know what you doing.
153 ```
154 # list
155 docker volume ls -qf dangling=true
156 # remove
157 docker volume rm $(docker volume ls -qf dangling=true)
158 ```
